# Версионирование релизов (SemVer)

Связано: [[00 — Главная (MOC)]], [[01 — Обзор проекта и цели MVP]], [[11 — Дорожная карта до MVP (фазы)]]

## Источник правды

Версия продукта в репозитории: поле **`version`** в `package.json` (дублируется в UI). История изменений по релизам — **`CHANGELOG.md`** в корне репозитория.

## Перед каждым коммитом

1. Поднять **patch** (третье число), оставаясь в той же **фазе** (minor):  
   `npm run version:patch`  
   (скрипт вызывает `npm version patch --no-git-tag-version` — только меняет `package.json` / lock, без тега git.)
2. Добавить запись в **`CHANGELOG.md`** под текущую версию.
3. При необходимости обновить **README**, строку в [[00 — Главная (MOC)]], этот файл или [[11 — Дорожная карта до MVP (фазы)]].

Переход на новую **фазу** (новый **minor**, например `0.1.x` → `0.2.0`) — вручную выставить версию в `package.json` и описать в `CHANGELOG`, когда выполнены критерии фазы в дорожной карте.

## Правила (договорённость)

### До MVP (`0.y.z`)

Формат: **`0.ФАЗА.ПАТЧ`**

| Часть | Смысл |
| ----- | ----- |
| **0** | Пре-MVP; пока не выполнен scope [[01 — Обзор проекта и цели MVP]]. |
| **ФАЗА** (второе число, **minor**) | Номер **фазы** дорожной карты — см. [[11 — Дорожная карта до MVP (фазы)]]. Повышается при выполнении критериев выхода из фазы. |
| **ПАТЧ** (третье число) | Итерации **внутри одной фазы**: фиксы, мелкие дополнения; перед коммитом увеличивать по правилам выше. |

Примеры: **`0.1.1`** — фаза 1, патч 1; **`0.2.0`** — начало фазы 2.

### Релиз MVP

| Версия   | Смысл |
| -------- | ----- |
| **1.0.0** | **Первый релиз MVP** — playable по Must Have из ТЗ (git-тег `v1.0.0`). |

После **1.0.0** — классический SemVer (major / minor / patch по смыслу изменений).

## Связь с ТЗ

ТЗ документа (`v1.0`) описывает дизайн; **1.0.0 кода** означает «реализован MVP по этому ТЗ», а не версию документа.
